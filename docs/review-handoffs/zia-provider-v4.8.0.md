# ZIA Provider 4.8.0 Review Handoff

## Intent

- Refresh only the ZIA pack and checked-in provider schema from `4.7.26` to
  `4.8.0`.
- Admit six existing-resource scalar fields that now have matching provider
  schema, Read, and expand evidence: `eun_template_id`,
  `exclude_context_shield_end_point`, and `is_eun_enabled` on both firewall DNS
  and firewall filtering rules.
- Preserve adoption safety for the new endpoint-application blocks by dropping
  them explicitly where provider readback is broken or unwired.
- Keep ZPA, the AST/source-analysis implementation, and new ZIA resource
  onboarding unchanged.

## Expected Delta

- Expected behavior change: firewall DNS/filtering transforms retain
  `eun_template_id`, `exclude_context_shield_end_point`, and `is_eun_enabled`.
  Endpoint application/application-group API fields remain absent from
  transformed and adopted configuration for DNS, filtering, IPS, and SSL.
- Expected report/count/coverage changes: six provider-missing holds are
  removed; four DNS/filtering endpoint holds are reclassified; two IPS
  endpoint holds are added. No readiness or OpenAPI coverage claim changes.
- Expected generated-output changes: the fresh provider schema adds nine
  resources and ten data sources. Existing resource deltas are DNS/filtering
  scalar and endpoint fields, IPS endpoint fields plus `eun_template_id`
  becoming Optional-only, SSL endpoint fields, and DLP web-rule `action`
  becoming Optional-only.
- Expected no-op areas: the nine new resource types are not in the ZIA registry
  and therefore are not collection/adoption/module targets. ZPA and AST work
  are unchanged.

## Invariants Claimed

- Evidence must not be silently dropped: endpoint exclusions moved to explicit
  `drops`; their unresolved provider behavior remains named in shared holds.
- Generic matcher evidence must not outrank source-backed evidence: no matcher
  or source-operation changes are included.
- Source precedence/provenance must remain explicit: provider and SDK tags,
  commits, schema digest, and fixture evidence links are versioned.
- Ambiguity must stay classified instead of being coerced to success: the new

- Provider-readiness counts must stay explainable: only raw provider schema
  inventory counts change; no new registry readiness status is asserted.
- Adoption safety invariants: the sandbox order-127 and URL-filtering ISOLATE

  files are byte-identical to `4.7.26`.

## Known Deferrals

- Deferred work: onboard the nine new ZIA resources and ten data sources.
- Reason it is safe to defer: they are schema-visible but absent from the pack
  registry, so this change makes no collection or adoption claims for them.
- Follow-up owner or trigger: a separate provider-resource adoption branch with
  API mappings, fixtures, source witnesses, and live round-trip evidence.
- Deferred work: endpoint application/application-group support on DNS,
  filtering, IPS, and SSL.
- Reason it is safe to defer: explicit top-level drops preserve prior adoption
  output. DNS/filtering/SSL resource Read calls the full data-source flatteners,
  whose object shapes do not fit the resource blocks; a direct
  `ResourceData.Set` probe returned invalid-address errors for applications and
  a type-conversion panic for groups. IPS declares the blocks but neither
  expands nor reads them.
- Follow-up owner or trigger: an upstream provider fix plus a live
  create/import/read/no-op-plan proof. Schema-derived modules still expose the
  optional provider-declared inputs; consumers must not set them before that
  proof.

## Review Focus

- Highest-risk files or paths: the regenerated ZIA schema, four rule overrides,
  shared adoption holds, provider-versioned registry guards, and generated
  compatibility fixtures.
- Specific assumptions to attack: only the six scalar holds are safe to close;
  explicit endpoint drops occur before schema projection; version-scoped
  sandbox and ISOLATE guards still match `4.8.0`; no new resource became
  registry-enabled accidentally.
- Source evidence the reviewer should verify: resource schema declarations,
  direct scalar `d.Set` and expand assignments, endpoint flattener call sites
  and shapes, absent IPS endpoint read/expand assignments, and unchanged
  sandbox/URL-filtering source files.
- Generated artifacts the reviewer should compare: reproduce the provider
  schema digest; verify the 83/115 counts and exact nine/ten additions; inspect
  parity and module compatibility drift for provenance-only or expected schema
  changes.
- Edge cases that could silently overclaim, remap, drop, or weaken evidence:
  raw camelCase endpoint objects versus the single-block set schema; optional
  module inputs despite adoption-time drops; stale provider-version predicates;
  treating schema presence as round-trip proof; accidental ZPA or AST drift.
